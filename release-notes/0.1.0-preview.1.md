# OpenBao Secret Sync 0.1.0-preview.1

This is the first preview release of the OpenBao Secret Sync secret engine
plugin. It is intended for evaluation, integration testing, and design
feedback. It is not a production-ready release or a compatibility boundary.

## Compatibility and support boundary

- OpenBao `2.6.0` is the minimum supported server and the only version
  qualified for this preview.
- OpenBao `2.5.x` is unsupported; later OpenBao releases remain unqualified
  until tested.
- APIs, response fields, storage records, provider behavior, ownership
  metadata, defaults, and operational guidance can change before a stable
  release.
- Upgrade, downgrade, and migration compatibility between preview releases is
  not promised.

### Development-build cleanup

Development builds before this preview wrote `plugin-instance` ownership
metadata to remote objects. The preview uses OpenBao's mount UUID instead and
does not adopt the earlier metadata. Remove those remote objects with the old
build or through the destination provider before upgrading; otherwise the new
build fails closed with an ownership conflict until the objects are removed and
recreated.

## Qualified provider scope

- AWS Secrets Manager supports JSON `secret-path` associations through the AWS
  SDK default chain, STS assume role, and STS web identity. The provider has
  LocalStack coverage plus opt-in real-AWS assume-role and web-identity
  qualification.
- Kubernetes Secrets supports JSON `secret-path` associations in the
  `data.payload` key and top-level `source-keys` data mapping. The provider is
  qualified against a real Kubernetes API in kind.
- GitLab supports project CI/CD variables, including association-level
  environment scope, protection, masking, hiding, expansion, and variable type
  configuration. The provider is qualified against Dockerized GitLab CE.
- Release binaries and the OCI plugin distribution are registered and mounted
  in OpenBao during the release workflow. The release binary is exercised
  against LocalStack, and the OCI image follows the OpenBao auto-download and
  auto-registration path.
- Native OpenBao create/update ACL semantics are enforced for source data,
  source metadata, destinations, and associations. Create-only policies cannot
  overwrite existing resources, and update-only policies cannot create missing
  resources.

## Known limitations

- Source data belongs to the Secret Sync plugin mount; existing KV mounts are
  not used as sync sources.
- GitLab group and instance variables are not supported in this release.
- AWS destination health uses `secretsmanager:ListSecrets`, which requires
  `Resource: "*"`. AWS value-drift detection requires opt-in value readback and
  `secretsmanager:GetSecretValue`.
- Queue dispatch is durable and bounded but currently sequential per active
  plugin process.
- Ownership metadata is stored in destination-native metadata carriers. It can
  expose source names, versions, object identifiers, payload hashes, and plugin
  mount and restore identity to principals that can inspect destination
  metadata.
- The project does not yet publish a stable support window.

Verify checksums, signatures, attestations, SBOMs, provenance, and OCI image
identity before installation by following
